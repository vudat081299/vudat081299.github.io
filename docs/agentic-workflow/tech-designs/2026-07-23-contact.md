## 1. Tổng quan

### Vấn đề
Đối tác vay/mượn hiện là free-text (`Loan.counterparty: string`, `types.ts:236`) — không tái dùng được, đổi tên một nơi không lan ra, không gom nhóm được. Cần một entity `Contact` có **định danh ổn định** để bản ghi khác tham chiếu bằng id thay vì chép tên. Đây là **slice A** (nền cho slice B loan redesign); slice A CHƯA nối loan vào contact.

### Goals
- Thêm entity `Contact` là first-class trong `CashyState`, đi qua đúng bộ máy store / persistence / migration / import-export như mọi entity khác.
- Đủ CRUD + vòng đời archive + màn quản lý riêng, phủ 12 BR + 1 NFR của spec.
- Để lại một "seam" tham chiếu (`isContactReferenced`) để slice B cắm vào mà không sửa lại kiến trúc.

### Non-Goals
- KHÔNG nối `Loan`/`Transaction` vào `Contact` (field FK + migration đó là **slice B**).
- KHÔNG tính số dư/“ai nợ tôi” theo contact (derived ở slice B).
- KHÔNG dựng repository/BE thật — Cashy 100% localStorage; chỉ giữ seam sẵn sàng cho BE.
- KHÔNG thêm phone/email/nhóm/merge/avatar-ảnh (YAGNI, theo spec Out-of-Scope).

## 2. Bối cảnh & Kiến trúc tổng quan

### Hiện trạng
- Kiến trúc phân lớp `ui → usecases → domain (+ data)`, enforce bởi `scripts/check-layers.mjs`. `domain/**` thuần, `data/**` là persistence seam.
- Mỗi entity là một mảng trong `CashyState` (`domain/types.ts:255`): `wallets`, `loans`, `subscriptions`… Contact sẽ là `contacts: Contact[]` — bản sao khuôn của `Wallet`/`Loan`.
- Persistence seam: `data/store.ts` (`getState`/`commit`), `data/persistence.ts` (`load`/`save`/`emptyState`), `data/migrations.ts` (`CURRENT_VERSION=8`, migrate append-only). Usecases đọc `getState()`, ghi `commit(next)`.
- Seed: `data/seed.ts` (seed rỗng) + `data/sample.ts` (`buildSampleData`, demo). Import/export: `usecases/workspace.ts`.
- Routing: hash-based, `lib/router.ts` (`Route` union + `ROUTES`), `App.tsx` switch theo route, `ui/app/Layout.tsx` nav list. Precedent màn hình: `ui/features/wallets/Wallets.tsx` (PageHeader + grid card + inline editor modal).
- UI card: molecule `CardIdentity` + class `.cashy-card*` (vừa chuẩn hoá ở commit `cd399bf`).

### Kiến trúc tổng quan (high-level / system context)
`N/A — gói trong 1 app, thay đổi additive theo đúng pattern entity sẵn có (wallet/loan). Không có service/integration ngoài.`

## 3. Giải pháp đề xuất

### Quyết định kiến trúc

- `ADR-contact-001`
  - **Quyết định**: `Contact` là entity top-level `CashyState.contacts: Contact[]`, không lồng trong entity khác.
  - **Lý do**: đồng nhất với mọi entity hiện có → tái dùng nguyên bộ store/persistence/migration/import-export, tích hợp gần như O(1).
  - **Trade-off chấp nhận**: thêm 1 mảng vào state toàn cục (không đáng kể; đúng khuôn đang dùng).

- `ADR-contact-002`
  - **Quyết định**: Logic nghiệp vụ đặt ở module thuần mới `domain/contact.ts` — chuẩn hoá + validate (`normalizeContactInput`, giới hạn tên ≤80 / username ≤30), lọc active (`activeContacts`), nhãn hiển thị (`contactLabel`), và **guard tham chiếu `isContactReferenced(state, id)`**.
  - **Lý do**: business rule thuộc `domain` (thuần, test được), không rải ở usecase/UI. Gom guard vào MỘT hàm để slice B chỉ mở rộng một chỗ.
  - **Trade-off chấp nhận**: `isContactReferenced` ở slice A luôn trả `false` (chưa ai tham chiếu) — một hàm "ngủ", chấp nhận vì nó khoá hợp đồng cho slice B.

- `ADR-contact-003`
  - **Quyết định**: `usecases/contacts.ts` CRUD (`addContact`→id, `updateContact`, `setContactArchived`, `deleteContact`) mirror `usecases/wallets.ts`; `deleteContact` gọi `isContactReferenced` — bị tham chiếu thì từ chối xoá cứng.
  - **Lý do**: usecase = orchestration trên store; guard nằm ở domain giữ tính thuần.
  - **Trade-off chấp nhận**: nhánh "từ chối xoá" chưa kích hoạt được ở slice A; vẫn hiện thực để hợp đồng ổn định.

- `ADR-contact-004`
  - **Quyết định**: Migration **v9** additive (`next.contacts ?? []`), `CURRENT_VERSION → 9`; seed demo contact trong `buildSampleData`; `emptyState`/`load`(nhánh re-seed)/`createWorkspace`/`exportData`/`importData` đều mang `contacts`.
  - **Lý do**: đúng khuôn v7 (loans mới → chỉ đảm bảo field tồn tại); wipe/reseed tự động qua đường seed sẵn có.
  - **Trade-off chấp nhận**: slice B (loan redesign) dời sang **v10** (memory `cashy-loan-redesign` ghi tạm v9 — cập nhật lại).

- `ADR-contact-005`
  - **Quyết định**: GIỮ `store.ts` + `persistence.ts` làm seam persistence/“query” duy nhất, BE-ready — KHÔNG thêm interface repository cho Contact.
  - **Lý do**: đáp directive tách UI/logic + lớp query, nhưng chỉ có MỘT implementation (localStorage); thêm interface cho một impl = shallow module test-past-seam (codebase-design). BE tương lai chỉ đổi `persistence.load/save` sang async, `usecases/domain/ui` không đổi.
  - **Trade-off chấp nhận**: chưa có "repository" tường minh; khi lên BE phải refactor persistence — rẻ hơn nhiều so với ôm abstraction sớm.

- `ADR-contact-006`
  - **Quyết định**: `ContactCard` tái dùng molecule `CardIdentity` + class `.cashy-card*`; màn `Contacts` mirror cấu trúc `Wallets` (PageHeader + grid + inline editor modal). Contact có `colorHex` + `icon` (như Wallet) để tile tô màu được.
  - **Lý do**: theo đúng convention card vừa chốt; không đẻ primitive UI mới. Giải quyết OQ "visual identity" bằng cách theo tiền lệ Wallet.
  - **Trade-off chấp nhận**: contact "nặng" thêm 2 field trình bày — nhất quán toàn app, chấp nhận.

### Data Model / Database Changes
Không có SQL. Thêm interface (integer/ISO string như các entity khác):

```ts
// domain/types.ts
export interface Contact {
  id: string;          // uid(), sinh 1 lần, BẤT BIẾN (BR-contact-005)
  name: string;        // bắt buộc, đã trim, 1..80 (BR-contact-001, 011)
  username?: string;   // tuỳ chọn, đã trim, ..30 — handle/định danh tài khoản (BR-contact-003, 012)
  colorHex: string;    // hue phân loại (như Wallet) — cho CardIdentity
  icon: string;        // curated lucide key
  archived: boolean;   // true = ẩn khỏi active selection, giữ bản ghi (BR-contact-006)
  createdAt: string;   // ISO
}
// CashyState thêm: contacts: Contact[];
```

Migration v9 (append-only, mirror v7): `if (fromVersion < 9) next = { ...next, contacts: next.contacts ?? [] };` + `CURRENT_VERSION = 9`. Không back-fill (không có dữ liệu cũ để chuyển). **Lý do** không tách bảng/không normalize: state là một cây JSON localStorage, mọi entity đều là mảng phẳng — thêm mảng là đủ.

## 4. Các phương án đã cân nhắc

- **Phương án đề xuất**: Contact là entity top-level theo đúng khuôn Wallet/Loan (state array → domain → usecase → migration → seed → import/export → screen). Thắng vì: rẻ nhất, nhất quán, tái dùng toàn bộ hạ tầng + convention card.
- **Đã loại — Contact như "wallet ảo" (liability wallet)**: nhét contact vào mô hình wallet. Loại vì: sai ngữ nghĩa (contact không giữ tiền ở slice A), làm bẩn `Wallet` union + các phép tính số dư.
- **Đã loại — Repository/interface `ContactStore` cho BE-ready**: trừu tượng hoá query ngay. Loại vì: chỉ một impl (localStorage) → shallow module, vi phạm codebase-design; seam `persistence.ts` đã đủ BE-ready (ADR-contact-005).
- **Đã loại — dời `ContactPicker` sang hẳn slice B**: gọn slice A hơn. Loại vì: BR-contact-004 (tạo inline khi gán) thuộc spec slice A + có scenario; build picker ở A giữ đúng phạm vi spec, B chỉ việc consume.
